Etebase Deep Dive

Can Etebase replace JobDone's planned privacy-first Local Replica protocol, or should it stay prior art?

Research date: 2026-06-07 Closest prior art Verdict: study, do not adopt wholesale

Short verdict

Etebase is close enough that we should study it before building encrypted payload mode. It does not replace Local Replica wholesale: JobDone still needs Sync Intents, Server T, Sync Partitions, private graph handling, Recall coverage, and non-password-rooted unlock.

Reuse stance

Use Etebase as prior art now. Run a throwaway spike before doing encryption. Do not block ordinary sync/Recall cleanup on it.

What Etebase Solves

Encrypted app data

Server stores encrypted Collections, Items, and Revisions. Client owns crypto and app content.

Sharing

Public-key collection invitations, member listing, access levels, revocation, and leaving collections.

Sync tokens

Collections expose `stoken` values for efficient changed-data fetches.

Consistency checks

Transactions and `stoken` checks can fail stale writes instead of overwriting silently.

Gaps Against Our Draft

  • Server T: Etebase uses collection sync tokens, not one global Datomic-like transaction cursor.
  • Sync Intents: business actions like claims, approvals, and invite acceptance still need an app-level intent layer.
  • Recall coverage: Etebase does not appear to expose "local search is complete for this query" as a first-class concept.
  • Private graph: encrypted payloads help, but access/fetch patterns can still leak relationship/query intent.
  • Sync Partitions: collection type tokens are useful prior art, but cold-restore coverage still needs app design.
  • Auth UX: current Etebase model is password-rooted; JobDone wants auth and unlock separable and eventually passkey-friendly.

Options

Option Pros Cons Verdict
Adopt Etebase wholesale E2EE/sharing already built. Auth mismatch, AGPL server, stale JS, no explicit coverage/intents. Not now.
Use as prior art Low disruption, keeps property tests and Server T direction. More code to build; later security review needed. Recommended.
Fork/adapt Builds on serious work. High maintenance, AGPL implications, security-sensitive fork. Only if sync infra becomes product.
Use as encrypted blob store Reuses encrypted storage. Two sync systems, hard tests, more failures. Avoid.

Recommended next move

Before encrypted payload implementation, run a throwaway Etebase spike: fake Entry/Contact/Location/Attachment, share with second user, test browser CSP/WASM, and see if collection/item fetch can support complete Recall coverage.
