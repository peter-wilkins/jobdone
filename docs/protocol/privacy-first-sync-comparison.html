Privacy-First Sync Protocol Prior Art Matrix

A comparison between JobDone's draft privacy-first sync protocol and existing E2EE, local-first, and managed sync systems. The goal is to decide whether to reuse, fork, study, or keep designing.

Draft 0 Research date: 2026-06-07 Not a novelty claim

Closest match

Etebase is closest for central-server E2EE app-data sync. Study it before building encrypted payload mode.

Most ambitious

any-sync is already a serious P2P, E2EE, local-first collaboration protocol. It is likely too much machinery for our MVP.

Our useful niche

Coverage-aware encrypted local recall: the app must know whether local search has all required data before saying "nothing found".

Has someone already done it?

Mostly yes for pieces. Etebase is close enough that it should be studied before building encryption. any-sync is close enough for P2P collaboration that we should not reinvent that path. The remaining niche is narrow: server-assisted, encrypted, typed app-data sync with explicit local query coverage.

What seems distinctive

  • Central server as ordering, authorization, durability, and restore infrastructure, not content reader.
  • Typed app objects through Sync Intents and Server T, not generic database row mirroring.
  • Coarse Sync Partitions for encrypted cold restore.
  • Private Relationship Graph hidden from backend by default.
  • Explicit Recall/query coverage so local search does not silently lie.
  • Implementation-independent design that can run on Postgres first.

Recommended next research move

Do a one-page Etebase deep dive: current maintenance, license, server deployability, browser/mobile client fit, collection-sharing model, metadata leakage, and whether its sync token model can express query coverage.
